01
The Past

Trước khi có REST: SOAP/XML

Thập niên 90: Dữ liệu bị gói trong những thẻ XML cồng kềnh.

<!-- Request Mua khóa học -->
<soap:Envelope xmlns:soap="...">
  <soap:Body>
    <m:BuyCourseRequest>
      <id>101</id>
    </m:BuyCourseRequest>
  </soap:Body>
</soap:Envelope>
Heavy

Tốn băng thông vì quá nhiều thẻ thừa.

Hard to Read

Khó đọc, khó debug cho con người.

03
Concept

Tư duy hướng Tài nguyên (Resource-Oriented Design)

Mọi thứ trên Web là một "Tài nguyên" (Resource).

Verb Resource URL Action
GET /courses Lấy danh sách khóa học
POST /courses Tạo mới một khóa học
GET /users/123 Lấy thông tin User ID 123
DELETE /courses/10 Xóa khóa học ID 10

06
The Problem

Giới hạn của REST

Bối cảnh Mobile (3G/4G) làm lộ rõ điểm yếu của sự cứng nhắc.

Unstable Network

Mạng chập chờn, băng thông thấp.

Vấn đề cốt lõi:
  • Cấu trúc dữ liệu trả về bị "đóng đinh" (Fixed).
  • Không thể tùy chỉnh theo màn hình thiết bị.
07
Problem 1

Vấn đề: Lấy thừa (Over-fetching)

Mobile Screen Cần:
  • Title
  • Price
// REST trả về tất cả:
{
  "title": "Java",
  "price": 10$,
  "desc": "Bài văn 1000 chữ...",
  "created_at": "2023...",
  "author_bio": "..."
}

Lãng phí băng thông 3G của người dùng.

08
Problem 2

Vấn đề: Lấy thiếu (Under-fetching)

Cần hiển thị: Course + Author + Comments.

GET /course/1
GET /user/99
GET /comments

N+1 Problem

Độ trễ (Latency) x3

09
The Solution

GraphQL (2015)

"Hãy hỏi chính xác những gì bạn cần, tôi sẽ trả lời đúng như thế."

Lee Byron(Engineering lead), người công bố GraphQL tại React.js Conf 2015 cùng Nick Schrock(Lead architect) và Brandon Krieg(Contributor chính trong giai đoạn open source)

10
GraphQL Concept

Phép màu của Single Endpoint

Chỉ có đúng 1 URL duy nhất: /graphql

# Client Query
query {
  course(id: 1) {
    title
    price
    instructor {
      avatar
    }
  }
}

Client toàn quyền quyết định dữ liệu.

11
Result

Phản hồi chính xác (The Precise Response)

Server trả về đúng cấu trúc yêu cầu. Không thừa 1 byte.

# Query (Input)
...
  title
  price
...
// JSON (Output)
{
  "data": {
    "course": {
      "title": "Java",
      "price": 10
    }
  }
}
12
Benefit

Sự tự do cho Frontend

No more "Backend Please"

Frontend Dev không cần nhờ Backend sửa API khi muốn thêm trường dữ liệu.

Ứng dụng mạnh mẽ cho: Tìm kiếm & Lọc (Filter) phức tạp.

13
Comparison

REST vs. GraphQL: Khi nào dùng ai?

Tiêu chí REST GraphQL
Caching Tuyệt vời (HTTP Cache) Khó (Phải tự làm)
Complexity Đơn giản Phức tạp (Learning Curve)
Data Shape Cứng nhắc (Fixed) Linh hoạt (Flexible)
Use Case Public API, File Upload Complex Mobile App, Dashboard
14
Realtime

Bài toán Real-time

Nhu cầu: Chat, Thông báo, Live Class.

HTTP Problem

Server thụ động. Không thể tự nói "Này, có tin nhắn mới".

15
Old Solution

Polling: Hỏi liên tục

Client
1s: Có tin chưa? -> Chưa
2s: Có tin chưa? -> Chưa
3s: Có tin chưa? -> Chưa
Server

--> Server Overload & Lãng phí tài nguyên.

16
The Solution

WebSockets (2011)

Giao thức Full-duplex (Song công) qua kết nối TCP bền vững.

Thay vì gửi thư tay (HTTP)...

...Ta nối một đường dây điện thoại và giữ máy liên tục.

Persistent Connection

17
Mechanism

Cơ chế Handshake

GET /chat HTTP/1.1
Host: server.com
Upgrade: websocket
Connection: Upgrade

// Server Response:
HTTP/1.1 101 Switching Protocols

HTTP đóng lại -> WebSocket mở ra.

18
Mechanism

The Push Model (Đẩy)

User A (Send)

Server

User B (Receive)

Server chủ động PUSH tin nhắn ngay lập tức. Độ trễ ~0.
